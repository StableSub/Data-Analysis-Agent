# 데이터 분석 AI 에이전트 제품 요구사항 문서(PRD)

## 문서 목적

이 문서는 현재 프로젝트인 데이터 분석 AI 에이전트의 제품 방향, 핵심 사용자 흐름, 기능 범위, PoC 기준을 정리한다.
구현 세부와 현재 제약은 [현재 구현 기준선](./current-state-baseline.md), 향후 개선 순서는 [구현 로드맵](./roadmap.md)을 함께 본다.

## 프로젝트 개요

- 프로젝트 주제: 자연어 기반 데이터 분석 AI 에이전트
- 한줄 요약: 사용자가 CSV 데이터셋을 업로드하고 질문하면, 시스템이 데이터 상태를 파악한 뒤 전처리, 분석, RAG/guideline 검색, 시각화, 리포트 생성을 조합해 답변하는 보조형 데이터 분석 플랫폼

## 문제 정의

비전문 사용자는 데이터셋을 가지고 있어도 어떤 전처리가 필요한지, 어떤 분석 방법을 써야 하는지, 결과를 어떻게 해석해야 하는지 판단하기 어렵다.
또한 분석 과정은 데이터 업로드, 품질 확인, 코드 작성, 결과 검증, 차트 생성, 리포트 작성처럼 여러 단계로 나뉘어 있어 반복 작업과 오류 가능성이 크다.

핵심 비효율은 아래와 같다.

- 데이터 구조와 품질을 직접 확인해야 한다.
- 자연어 질문을 실제 분석 계획과 실행 코드로 옮기기 어렵다.
- 분석 결과의 근거, 사용 컬럼, 실행 방식이 흩어져 추적하기 어렵다.
- 시각화와 리포트 작성이 분석 결과와 분리되어 반복 작업이 발생한다.
- 실패한 분석을 디버깅하거나 재현하기 어렵다.

## 서비스 대상과 가정

### 핵심 사용자

- 데이터 분석 경험이 적은 학생, 연구자, 실무자
- CSV 기반 데이터셋을 빠르게 탐색하고 질문하고 싶은 사용자
- 분석 결과를 차트나 리포트 형태로 정리해야 하는 사용자

### PoC 가정

- 초기 입력 데이터는 CSV 중심으로 제한한다.
- 사용자는 Workbench UI에서 데이터셋을 업로드하고 질문한다.
- AI는 모든 결정을 완전 자동화하지 않고, 전처리·시각화·리포트처럼 사용자의 확인이 필요한 단계에서 approval/resume 흐름을 사용한다.
- 외부 문서 근거가 필요한 경우 guideline/RAG 검색을 보조 컨텍스트로 사용한다.

## 제품 목표

1. 사용자가 데이터셋을 업로드하고 구조를 확인할 수 있다.
2. 사용자의 자연어 질문을 분석 가능한 계획으로 변환한다.
3. 데이터 품질에 따라 전처리 필요성을 제안하고 사용자의 승인을 받는다.
4. SQL 또는 Python 기반 분석 실행 결과를 생성하고 저장한다.
5. 필요한 경우 RAG/guideline 근거를 검색해 답변 맥락을 보강한다.
6. 분석 결과를 시각화하고 리포트 초안으로 확장한다.
7. 실행 단계, 승인, 오류, 최종 결과를 trace/logging으로 추적한다.

## 핵심 사용자 흐름

1. 사용자가 CSV 데이터셋을 업로드한다.
2. 시스템이 데이터셋 메타데이터, 샘플, 프로파일을 생성한다.
3. 사용자가 데이터셋을 선택하고 자연어 질문을 입력한다.
4. workflow가 질문과 dataset 선택 여부를 기준으로 실행 경로를 결정한다.
5. 전처리가 필요하면 계획을 제안하고 사용자 승인을 기다린다.
6. 분석 또는 RAG 경로가 실행되고 결과 컨텍스트가 누적된다.
7. 필요 시 guideline 검색, 시각화, 리포트 생성이 이어진다.
8. 시스템은 최종 답변과 결과 artifact를 SSE로 전달한다.
9. 실행 상태와 오류 정보는 trace/logging 구조에 남는다.

## 기능 요구사항

### 1. 데이터셋 관리

- CSV 데이터셋을 업로드할 수 있어야 한다.
- 업로드된 데이터셋의 목록, 상세 정보, 샘플 행을 조회할 수 있어야 한다.
- 데이터셋 삭제와 관련 인덱스 정리를 지원해야 한다.

### 2. 데이터 프로파일링과 EDA

- 컬럼, 타입, 결측치, 기본 통계, 분포 등 분석 전 정보를 제공해야 한다.
- 전처리 추천에 필요한 품질 정보를 생성해야 한다.

### 3. 자연어 분석 실행

- 사용자의 질문을 분석 목표, 사용 컬럼, 집계 방식, 실행 전략으로 변환해야 한다.
- 단순 집계는 SQL 실행 경로를 사용할 수 있어야 한다.
- 복잡한 분석은 안전한 Python sandbox 실행 경로를 사용할 수 있어야 한다.
- 실행 결과에는 요약, 표, raw metric, 사용 컬럼, 오류 정보가 포함되어야 한다.

### 4. 전처리 approval/resume

- 전처리가 필요한 경우 계획과 영향을 사용자에게 보여줘야 한다.
- 사용자는 승인, 수정 요청, 취소를 선택할 수 있어야 한다.
- 승인된 전처리는 새 output source로 저장되어 이후 분석에 사용되어야 한다.

### 5. RAG/guideline 근거 검색

- 데이터셋 또는 guideline 문서에 대해 검색 기반 컨텍스트를 만들 수 있어야 한다.
- 검색 결과와 evidence summary는 최종 응답 또는 리포트 생성에 활용될 수 있어야 한다.

### 6. 시각화와 리포트

- 분석 결과를 기반으로 적절한 차트 계획을 만들고 시각화 결과를 생성해야 한다.
- 리포트 초안을 생성하고 approval/revision/finalize 흐름을 지원해야 한다.

### 7. 세션, 스트리밍, trace

- 질문 실행은 SSE로 진행 상태와 최종 결과를 스트리밍해야 한다.
- 세션 히스토리와 pending approval 상태를 조회할 수 있어야 한다.
- `trace_id`, `session_id`, `run_id`, `stage`를 기준으로 실행 이력을 추적할 수 있어야 한다.

## 대표 시나리오

### 시나리오 1: 데이터셋 업로드 후 기본 분석

사용자가 매출 CSV를 업로드하고 “지역별 평균 매출을 알려줘”라고 질문한다.
시스템은 컬럼을 grounding하고 SQL 집계 계획을 만든 뒤 결과 표와 요약을 반환한다.

### 시나리오 2: 전처리 승인 후 분석

사용자가 결측치가 많은 데이터셋을 선택하고 분석을 요청한다.
시스템은 전처리 계획을 제안하고, 사용자가 승인하면 전처리된 output source를 만든 뒤 분석을 이어간다.

### 시나리오 3: 분석 결과 기반 시각화와 리포트

사용자가 분석 결과를 차트와 리포트로 정리해 달라고 요청한다.
시스템은 시각화 계획을 만들고 필요한 경우 승인을 받은 뒤, 리포트 초안을 생성해 최종 결과로 제공한다.

## PoC 우선 구현 범위

### 데이터 입력

- CSV 업로드
- 데이터셋 목록/상세/샘플 조회
- 데이터셋 삭제

### 분석 흐름

- 자연어 질문 입력
- dataset-aware workflow 실행
- 전처리 제안과 approval/resume
- 분석 계획 수립과 실행
- RAG/guideline 보조 근거 검색
- 시각화 생성
- 리포트 생성

### Workbench 화면

- 데이터셋 업로드와 선택
- 질문 입력과 SSE 진행 상태 표시
- approval card 표시와 재개
- 분석/시각화/리포트 결과 패널
- 세션 복원

## 우선순위

### 필수 항목

- CSV 업로드와 데이터셋 선택
- dataset-aware chat stream 실행
- 분석 계획과 실행 결과 생성
- 전처리 approval/resume 흐름
- 기본 RAG/guideline 검색
- 시각화 결과 생성
- trace/logging 기반 실행 추적

### 권장 항목

- 분석 정확도 벤치마크 하네스
- RAG evidence 품질 검증
- stage latency와 trace chain 표시
- 오류 원인 분류와 repair loop 개선
- 프롬프트 버전 관리

### 선택 항목

- CSV 외 표 형식 지원
- 복수 데이터셋 비교 분석
- 프롬프트/모델 A/B 평가
- 더 풍부한 visualization renderer

## 제외 범위

- 완전한 BI 도구 수준의 대시보드 빌더
- 모든 파일 형식의 범용 분석 지원
- 사람 검토 없는 위험한 데이터 변환 자동 적용
- 운영 배포용 권한/조직 관리

## 성공 기준

- 사용자가 데이터셋을 업로드하고 선택할 수 있다.
- 시스템이 자연어 질문을 분석 경로로 연결할 수 있다.
- 분석 결과가 표/요약/metric 형태로 반환된다.
- 필요한 단계에서 approval/resume이 동작한다.
- 시각화와 리포트 artifact를 생성할 수 있다.
- 실행 과정이 trace/logging으로 재현 가능하다.
