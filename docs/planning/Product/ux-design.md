# 사용자 경험 설계

## 문서 목적

이 문서는 사용자가 제품을 어떤 흐름으로 사용하고 어떤 경험을 하게 되는지 설명한다.
현재 기준에서는 사출성형 데이터 이상 진단 시나리오를 대표 사례로 삼아, 사용자의 진입부터 결과 확인까지의 경험을 정리한다.

## UX 설계 목표

이 제품의 사용자 경험은 복잡한 분석 도구를 직접 다루기 어려운 사용자도 자연어 기반으로 데이터를 해석하고 결과를 이해할 수 있도록 설계하는 것을 목표로 한다.

이를 위해 아래 세 가지를 UX의 기본 원칙으로 둔다.

- 사용자는 먼저 질문하고, 시스템은 그 질문을 분석 가능한 단계로 풀어준다.
- 분석 과정에서 사용자가 판단해야 하는 지점은 명확하게 분리한다.
- 결과는 단순 수치 나열이 아니라, 상태 판단과 다음 행동이 함께 보이도록 정리한다.

## 기본 사용자 여정

사용자 경험의 기본 흐름은 아래와 같다.

1. 사용자가 데이터셋 또는 분석 대상을 선택한다.
2. 사용자가 자연어로 분석 요청을 입력한다.
3. 시스템이 데이터를 불러오고 현재 상태를 요약해서 보여준다.
4. 필요 시 전처리 제안을 하고, 사용자가 승인 여부를 결정한다.
5. 시스템이 분석을 수행하고 이상 구간, 주요 변수, 원인 후보를 제시한다.
6. 시스템이 최종 결과를 요약하고 후속 조치 방향을 함께 안내한다.

이 흐름은 사용자가 분석 도구를 직접 조작하는 방식이 아니라, 질문과 승인 중심으로 진행되는 경험을 전제로 한다.

## 단계별 사용자 경험 설계

### 1. 진입과 분석 요청

사용자는 웹 대시보드에서 분석 대상 구간을 선택하거나, 이미 선택된 데이터셋을 기준으로 질문을 입력한다.
이 단계에서 중요한 경험은 “무엇을 분석할지”와 “어떤 목적의 질문인지”를 쉽게 표현할 수 있어야 한다는 점이다.

따라서 입력 영역은 복잡한 조건 설정 화면보다 자연어 요청 중심으로 구성하는 것이 적합하다.

### 2. 데이터 확인과 초기 이해

시스템은 분석을 시작하면 먼저 주요 변수의 추이와 분포를 보여주어 사용자가 현재 데이터 상태를 빠르게 이해할 수 있게 해야 한다.
이 단계의 목적은 정밀한 해석이 아니라, 사용자가 지금 어떤 구간과 어떤 변수들이 분석 대상이 되는지 감을 잡게 하는 것이다.

사용자는 이 시점에서 데이터가 정상적으로 불러와졌는지, 어떤 변수들이 핵심인지, 전반적으로 이상 징후가 있는지 초기 인상을 얻게 된다.

### 3. 전처리 제안과 사용자 승인

전처리가 필요한 경우, 시스템은 이를 자동으로 진행하기보다 먼저 이유를 설명하고 사용자에게 승인 여부를 묻는 방식이 적절하다.
사용자는 이 단계에서 분석 정확도를 높이기 위한 전처리의 필요성을 이해하고, 실제로 적용할지 결정한다.

따라서 UX는 아래 질문에 답하도록 설계되어야 한다.

- 왜 전처리가 필요한가
- 어떤 방식의 전처리가 적용되는가
- 전처리 전후에 무엇이 달라지는가

이 단계는 사용자가 개입하는 핵심 결정 지점이므로, 설명과 승인 인터페이스가 분리되어 있어야 한다.

### 4. 분석 결과 확인

분석 단계에서는 사용자가 가장 먼저 알아야 하는 내용을 우선 보여주는 것이 중요하다.
이 시나리오에서는 아래 순서가 적합하다.

1. 이상 상태 여부
2. 이상이 발생한 구간
3. 가장 관련성이 높은 변수
4. 원인 후보와 해석

즉, 사용자가 수많은 지표를 먼저 읽는 것이 아니라, “정상인지 아닌지”, “어디서 문제가 생겼는지”, “무엇을 먼저 봐야 하는지”를 빠르게 이해할 수 있어야 한다.

### 5. 원인 분석과 후속 조치 안내

이상 탐지 결과만으로 끝내지 않고, 관련 지식이나 매뉴얼을 연결해 원인 후보를 설명하는 경험이 필요하다.
사용자는 단순히 이상이 있다는 사실보다, 왜 그런 결과가 나왔고 무엇을 우선 점검해야 하는지를 알고 싶어한다.

따라서 결과 화면은 원인 후보, 관련 근거, 우선 점검 항목을 함께 보여주는 구조가 적합하다.

### 6. 최종 결과 정리

최종 결과는 분석 과정의 모든 세부 단계를 그대로 나열하기보다, 사용자가 바로 공유하거나 후속 조치에 활용할 수 있는 형태로 정리되어야 한다.
이 단계에서는 아래 요소가 함께 보이는 것이 중요하다.

- 현재 상태 요약
- 주요 이상 구간
- 핵심 원인 후보
- 우선 조치 항목

## 사용자가 결정해야 하는 지점

이 제품에서 사용자가 직접 결정해야 하는 지점은 많지 않아야 한다.
다만 아래와 같은 지점은 명확하게 사용자 판단이 들어가야 한다.

- 어떤 데이터 구간을 분석할지
- 어떤 질문을 던질지
- 전처리 제안을 승인할지
- 결과를 바탕으로 후속 조치를 진행할지

즉, 사용자의 결정은 “분석 로직의 세부 조정”이 아니라 “분석 대상과 실행 여부의 승인” 수준에 머무는 것이 바람직하다.

## UX 원칙과 주의사항

### 1. 질문 중심 경험 유지

사용자는 분석 도구를 조작하기보다 질문을 던지고 결과를 이해하는 방식에 익숙해야 한다.
따라서 인터페이스는 설정 화면보다 질문과 응답 흐름이 중심이 되어야 한다.

### 2. 승인 단계는 명확하게 분리

전처리와 같이 결과에 영향을 주는 단계는 자동 실행보다 승인 기반이 더 적합하다.
사용자가 지금 무엇을 승인하는지, 승인하면 어떤 변화가 생기는지 명확히 이해할 수 있어야 한다.

### 3. 결과는 해석 가능한 형태로 제공

사용자에게는 모델 내부 계산보다 현재 상태, 이상 구간, 원인 후보, 조치 방향이 더 중요하다.
따라서 결과 화면은 분석 과정 설명보다 해석과 의사결정 보조에 초점을 맞춰야 한다.

### 4. 복잡함은 시스템이 흡수

모델 선택, 전처리 방식, 분석 절차의 복잡함은 시스템이 내부적으로 처리하고, 사용자는 핵심 판단과 결과 이해에만 집중할 수 있어야 한다.
