# 시스템 플로우 개요

## 문서 목적

이 문서는 사용자 요청이 어떤 단계를 거쳐 최종 결과로 이어지는지 설명한다.
현재 기준의 시스템 흐름은 웹 기반 채팅 인터페이스를 통해 질문이 들어오고, 데이터 선택 여부와 intake의 coarse intent 결과에 따라 여러 분기 경로로 나뉘는 구조를 가진다.
이 문서는 기획, 프론트엔드, 백엔드가 함께 참고할 수 있도록 쉬운 언어로 설명하되, 흐름 자체는 현재 구현 기준과 어긋나지 않게 정리하는 것을 목표로 한다.

## 시스템 진입점

사용자 요청의 시작점은 `POST /chats/stream` 이다.
사용자가 질문을 보내면 시스템은 이 요청을 기준으로 하나의 실행 흐름을 시작한다.

실행 중 승인 대기 상태가 발생하면, 이후 흐름은 `POST /chats/{session_id}/runs/{run_id}/resume` 으로 이어서 재개된다.
사용자 입장에서는 하나의 대화처럼 보이지만, 시스템 내부에서는 `질문 시작`, `승인 대기`, `재개`가 구분되어 관리된다.

## 최상위 분기

시스템은 요청이 들어오면 가장 먼저 `general_question` 과 `dataset_selected` 중 어느 경로로 갈지 나눈다.

- `general_question`: 데이터셋이 선택되지 않은 일반 질문 경로
- `dataset_selected`: 데이터셋이 선택된 데이터 기반 처리 경로

즉, 데이터가 없는 질문은 바로 일반 답변으로 끝날 수 있고, 데이터가 있는 질문만 이후 분석 준비 단계로 이어진다.

## 데이터셋 기반 처리 흐름

데이터셋 기반 질문으로 들어오면 시스템은 아래 순서로 흐른다.

1. `preprocess_flow`
2. `analysis_flow` 또는 `rag_flow`
3. 필요 시 `guideline_flow`
4. 필요 시 `visualization_flow`
5. `merge_context`

이 구간은 아직 최종 답변을 생성하는 단계가 아니다.
시스템이 전처리 필요 여부를 판단하고, 분석 또는 검색 기반 설명 경로를 태운 뒤, 결과를 누적 컨텍스트로 합치는 단계에 가깝다.

## Intake 이후 분기

`intake_flow`는 질문을 받은 뒤 이후 경로의 큰 방향을 결정한다.
현재 구현 기준으로 intake 이후에는 아래 세 가지 최상위 경로가 존재한다.

- `general_question`
- `clarification`
- `data_pipeline`

각 분기의 의미는 다음과 같다.

- `general_question`: 데이터셋 없이도 답할 수 있는 일반 질문
- `clarification`: 추가 질문이 필요해 사용자에게 다시 물어보는 경우
- `data_pipeline`: selected-dataset 질문이 preprocess/analysis/rag/visualization/report 경로로 이어지는 경우

이 문서에서 중요한 점은, 현재 브랜치가 planner 하나로 세밀 분기를 하는 구조가 아니라 `handoff.ask_*` 플래그를 통해 후속 흐름을 조합하는 구조라는 점이다.

## 승인 필요 단계

현재 구현 기준으로 승인 흐름이 들어가는 대표 단계는 아래 세 가지다.

### 1. 전처리

전처리가 필요하다고 판단되면, 시스템은 전처리 계획을 제안하고 사용자의 응답을 기다린다.
사용자는 아래 중 하나를 선택할 수 있다.

- 승인
- 수정 요청
- 취소

승인 시 전처리가 실행되고, 수정 요청 시 계획을 다시 만들며, 취소 시 해당 흐름은 종료된다.

### 2. 시각화

시각화가 필요한 경우에도 시스템은 바로 실행하지 않고, 시각화 계획 검토 단계를 거친다.
이 단계에서도 사용자는 승인, 수정 요청, 취소를 선택할 수 있다.

### 3. 리포트

리포트 역시 초안 검토 단계를 거친다.
사용자는 리포트 초안을 보고 승인, 수정 요청, 취소 중 하나를 선택할 수 있다.

즉, 현재 시스템은 승인 구조를 전처리에만 한정하지 않고, 시각화와 리포트에도 적용하고 있다.

## 시각화와 리포트 분기

분석이나 설명형 질의응답 이후에는 추가 분기가 다시 발생할 수 있다.

### 시각화 분기

시각화는 `analysis` 뒤에도 갈 수 있고 `rag` 뒤에도 갈 수 있다.
즉, 시각화는 분석 결과만을 위한 전용 단계가 아니라, 데이터 기반 답변을 더 잘 이해하게 하는 보조 출력 경로다.

### 리포트 분기

시각화 여부와 별개로, `merge_context` 이후에는 `report` 또는 `data_qa`로 분기한다.
이 구조에서 리포트는 “최종 답변 뒤에 붙는 후속 기능”이라기보다, `data_qa`와 나란한 출력 경로에 가깝다.

즉, 시스템은 하나의 질문에 대해 항상 같은 형태의 결과를 내지 않으며, 맥락에 따라 대화형 답변 또는 리포트 형태의 결과 중 하나로 이어질 수 있다.

## 종료 경로

현재 구현 기준으로 시스템의 주요 종료 경로는 아래와 같다.

- `general_question_terminal`
- `clarification_terminal`
- `data_qa_terminal`
- `report_flow` 종료
- `cancelled`
- `failed`

따라서 이 문서는 성공 경로만 설명하는 문서가 아니라, 사용자의 질문이 어떤 분기와 종료 경로를 가질 수 있는지 전체 구조를 보여주는 문서로 이해하는 것이 맞다.

## 사용자가 받는 결과 유형

사용자는 질문 유형에 따라 아래 중 하나 또는 여러 개를 받는다.

- 일반 질문에 대한 텍스트 응답
- 데이터 분석 결과 요약
- 추가 설명이 필요한 경우의 clarification 질문
- 이상 구간과 주요 변수에 대한 설명
- 시각화 결과
- 리포트 형태의 정리된 결과물
- 실행 취소 또는 실패 안내

즉, 최종 결과는 하나의 고정된 형식이 아니라, 질문의 성격과 시스템 판단에 따라 달라진다.

## 이 문서를 읽는 방법

이 문서는 이상적인 목표 구조를 설명하는 문서가 아니라, 현재 구현 기준의 분기 구조를 설명하는 문서다.
따라서 이 문서를 읽을 때는 “질문 하나가 언제 어떤 경로로 갈 수 있는가”를 이해하는 것이 핵심이다.

즉, 이 시스템은 단순한 직선형 분석 파이프라인이 아니라, 질문 유형, intake 판단, 승인 여부, 시각화 및 리포트 필요성에 따라 여러 갈래로 나뉘는 분기형 시스템으로 이해하는 것이 맞다.
이 문서를 기준으로 이후의 시스템 아키텍처, 백엔드 구조, API 문서는 현재 구현 흐름과 연결되는 방식으로 이어서 읽는 것이 적합하다.
