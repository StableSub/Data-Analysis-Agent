# AI 코딩 정책

## 목적

이 문서는 캡스톤 발표 전까지 팀이 AI 도구를 사용할 때 지켜야 할 최소 운영 기준을 정리한다.
목표는 "빠르게 만든다"보다 "설명 가능하게 만든다"에 있다. 발표에서는 결과뿐 아니라 왜 그렇게 구현했고 어떻게 검증했는지까지 답해야 하므로, 모든 AI 활용은 재현 가능성과 책임 소재를 남겨야 한다.

## 이 정책이 적용되는 범위

- 코드 작성, 수정, 삭제
- 문서 초안 작성과 요약
- 테스트 케이스 초안 생성
- 발표 자료용 설명 문안 초안 작성
- 디버깅 아이디어, 원인 가설, 리팩터링 제안

## 핵심 원칙

1. **코드를 기준으로 말한다.**
   - AI가 설명한 내용보다 실제 repository와 실행 결과를 우선한다.
   - 특히 아키텍처, API, SSE, approval 흐름은 `backend/app/`, `frontend/src/`, `docs/architecture/`를 함께 확인한다.
2. **없는 검증을 있다고 말하지 않는다.**
   - 현재 프론트엔드에는 `lint`, `check-types` 스크립트가 없다.
   - 검증 결과를 보고할 때는 실제 실행한 명령만 적는다.
3. **작은 변경, 짧은 근거, 명확한 책임을 유지한다.**
   - 한 번에 너무 넓게 고치지 않는다.
   - 변경 이유, 영향 범위, 검증 결과를 PR/커밋/작업 기록에 남긴다.
4. **AI 출력은 초안일 뿐, 최종 책임은 사람에게 있다.**
   - merge 전에는 반드시 사람이 diff와 검증 로그를 읽는다.
5. **문서와 코드 drift를 방치하지 않는다.**
   - runtime 계약을 바꾸면 관련 문서도 같은 변경에서 갱신한다.

## 허용되는 AI 활용 방식

### 1. 초안 작성
- 함수/문서/테스트의 첫 초안을 빠르게 만든다.
- 단, 초안 채택 전에 naming, scope, side effect를 사람이 정리한다.

### 2. 탐색 보조
- 관련 파일 후보, 영향 범위, 유사 구현 패턴을 찾는 데 사용한다.
- "어디를 봐야 하는가"를 줄이는 용도로 쓰고, 근거 없는 대규모 변경은 금지한다.

### 3. 리뷰 보조
- 누락된 테스트, 문서 drift, 경계 조건, 발표 시 취약한 설명 지점을 점검하는 데 사용한다.
- 단, 리뷰 코멘트 채택 전에는 실제 코드/로그와 맞는지 확인한다.

## 금지 사항

- 실행하지 않은 테스트를 통과했다고 보고하기
- repo에 없는 스크립트나 CI 잡이 존재한다고 가정하기
- 사용자/심사위원에게 보여줄 설명을 코드 확인 없이 사실처럼 단정하기
- 근거 없는 전면 리팩터링, 폴더 구조 개편, 의존성 추가
- 다른 사람이 소유한 파일을 사전 합의 없이 넓게 수정하기
- AGENTS.md를 인간 팀 운영 문서 대신으로 취급하기

## 작업 전 체크리스트

- [ ] 이번 변경의 목적을 한 문장으로 설명할 수 있다.
- [ ] 어떤 파일을 바꿀지 미리 적었다.
- [ ] 관련 문서/테스트/런타임 경로를 확인했다.
- [ ] repo에 실제 존재하는 검증 명령만 사용할 계획이다.
- [ ] 발표 시 질문받을 위험 포인트를 한 가지 이상 생각했다.

## 작업 중 체크리스트

- [ ] AI가 제안한 코드/문장이 현재 코드 구조와 맞는지 확인했다.
- [ ] scope 밖 파일 수정이 생기면 이유를 기록했다.
- [ ] 사용자-facing 계약(API, SSE, payload, route, status code) 변경 여부를 점검했다.
- [ ] 문서가 바뀌어야 하면 같은 작업에서 함께 수정했다.

## 작업 후 체크리스트

- [ ] 변경 이유를 diff만 보지 않고도 설명할 수 있다.
- [ ] 실행한 검증 명령과 결과를 남겼다.
- [ ] 미검증 항목이 있으면 명시했다.
- [ ] 발표 자료나 데모 스크립트에 영향을 주면 관련 담당자에게 공유했다.

## 발표 전 특별 원칙

캡스톤 직전에는 "새로운 기능 추가"보다 "설명 가능한 안정성"이 우선이다.

- 발표 1주 전부터는 큰 구조 변경보다 재현성 확보를 우선한다.
- 데모에 쓰는 흐름은 문서, 스크립트, 검증 로그를 함께 묶어 관리한다.
- 정량 지표가 없는 주장은 발표에서 사용하지 않는다.
- 실패 가능성이 있는 기능은 숨기지 말고, 한계와 우회 전략을 함께 정리한다.

## 권장 보고 형식

작업 완료 보고는 아래 네 줄을 기본으로 한다.

1. 무엇을 바꿨는가
2. 왜 바꿨는가
3. 무엇으로 검증했는가
4. 아직 남은 위험은 무엇인가

이 네 줄이 빠지면 AI를 사용했더라도 팀 지식으로 축적되지 않는다.
