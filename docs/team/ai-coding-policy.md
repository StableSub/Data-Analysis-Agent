# AI 코딩 운영 정책

## 문서 목적

이 문서는 캡스톤 발표 전까지 팀이 AI 도구를 사용할 때 지켜야 할 공통 운영 정책을 정리한다.
루트 `AGENTS.md`가 AI 에이전트 실행 규칙이라면, 이 문서는 사람 팀원의 의사결정·검토·증빙 방식을 정의한다.

## 적용 범위

- 코드 작성·수정·리팩터링
- 문서 작성·정리·번역
- 발표 자료에 들어가는 기술 설명과 검증 근거 정리
- PR 작성, 리뷰, 병합 전 검증

## 운영 원칙

1. **코드와 실행 결과를 기준으로 판단한다.**
   - AI 설명보다 실제 저장소 파일과 검증 명령 결과를 우선한다.
2. **검증 없는 주장 금지.**
   - “동작한다”, “해결됐다”, “안전하다” 같은 표현은 명령 출력이나 화면 증거가 있을 때만 쓴다.
3. **작은 단위로 수정하고 바로 검증한다.**
   - 큰 범위의 추측성 변경보다, 목적이 분명한 작은 diff를 우선한다.
4. **발표 직전에는 재현 가능한 근거를 남긴다.**
   - 최종 데모 흐름, 테스트 명령, 확인 스크린샷 위치를 PR 또는 작업 기록에 함께 남긴다.
5. **AI를 작성 도구로 쓰되 책임은 사람이 진다.**
   - 최종 merge 판단, 발표 자료 반영, 대외 설명은 담당 사람이 책임진다.

## 허용 작업

- 반복적인 코드/문서 초안 작성
- 기존 패턴을 따르는 보일러플레이트 생성
- 테스트 시나리오 초안, 체크리스트 정리
- architecture/product 문서 정리
- 발표용 비교표·요약문 초안 작성

## 금지 작업

- 검증 없이 benchmark 수치나 정확도 지표를 만들어 넣는 것
- 존재하지 않는 CI, lint, typecheck 체계를 있다고 설명하는 것
- 실제 코드에 없는 workflow, route, event 이름을 문서나 발표 자료에 적는 것
- 비공개 키, 민감 데이터, 로컬 절대 경로를 외부 공유 문서에 넣는 것
- 사람 검토 없이 발표 핵심 주장(정확도, 안정성, 차별점)을 확정하는 것

## 작업 절차

### 1. 작업 시작 전

- 관련 범위의 `AGENTS.md`와 `docs/*` 기준 문서를 확인한다.
- 작업 목표를 한 문장으로 적는다.
- 수정 범위가 코드인지 문서인지 명확히 정한다.

### 2. 작업 중

- AI가 제안한 변경은 저장소의 실제 파일 경로와 맞는지 확인한다.
- 코드 수정 시 관련 테스트/빌드 명령을 바로 실행한다.
- 문서 수정 시 코드 경로, 링크, 용어를 실제 저장소 기준으로 맞춘다.

### 3. 작업 종료 전

- 아래 증빙 묶음을 남긴다.
  - 변경 파일 목록
  - 실행한 검증 명령
  - PASS/FAIL 결과
  - 발표 영향도(있음/없음)
  - 후속 확인 필요 사항

## PR 전 품질 게이트

자세한 게이트는 `../testing/quality-gates.md`를 따른다.
최소 기준은 아래와 같다.

- [ ] 변경 목적이 PR 설명 첫 줄에 드러난다.
- [ ] AI가 만든 설명을 그대로 복사하지 않고 팀 문체로 정리했다.
- [ ] 관련 검증 명령과 결과를 적었다.
- [ ] 발표 자료/데모에 영향이 있으면 체크했다.
- [ ] reviewer가 재현 가능한 파일 경로와 명령이 있다.

## 발표 직전 강화 규칙

- 데모에 쓰는 경로는 `main` 또는 발표용 브랜치 기준으로 다시 검증한다.
- “정량 평가”, “벤치마크”, “정확도”라는 표현은 `docs/testing/benchmark-spec.md` 기준 데이터가 있을 때만 사용한다.
- architecture 슬라이드에 적는 route/event 이름은 `docs/architecture/system/api-spec.md`, `docs/architecture/system/api-sse-error-contract.md`와 맞춰본다.
- 오류 대응 시나리오(예: `approval_required`, `error`)는 발표자가 직접 한 번 이상 리허설한다.

## 역할별 책임

자세한 소유권은 `./roles-and-ownership.md`를 따른다.

- 작성자: 변경 구현, 1차 검증, 증빙 정리
- 리뷰어: 주장-근거 일치 확인, 회귀 위험 확인
- 발표 담당: 발표 문구와 저장소 근거 일치 여부 최종 확인

## 빠른 체크리스트

### 작성자 체크리스트

- [ ] 이 변경이 왜 필요한지 한 줄로 설명할 수 있다.
- [ ] 실제 코드/문서 경로를 확인했다.
- [ ] 검증 명령을 실행했다.
- [ ] 과장된 표현이나 미구현 계획을 제거했다.

### 리뷰어 체크리스트

- [ ] diff 범위가 요청과 맞는다.
- [ ] 근거 없는 성능/정확도 주장이 없다.
- [ ] 발표 시 물어볼 만한 질문에 답할 근거가 남아 있다.
- [ ] 후속 작업이 있으면 명시되어 있다.
