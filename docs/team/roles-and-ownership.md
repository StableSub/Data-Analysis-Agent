# 역할과 소유권

## 목적

이 문서는 캡스톤 발표 준비 기간 동안 팀이 어떤 영역을 누가 책임지는지 빠르게 합의하기 위한 운영 문서다.
완벽한 조직도가 아니라, 작업 충돌과 책임 공백을 줄이는 최소 기준을 제공하는 것이 목적이다.

## 기본 원칙

- 소유권은 권한이 아니라 책임이다.
- 담당자는 직접 수정하지 않더라도 최종 설명 가능성을 가진다.
- 경계가 겹치는 영역은 공동 작업이 가능하지만, 최종 승인자는 한 명이어야 한다.
- 발표 직전에는 "다 같이 본다"보다 "누가 마지막 책임자인지 안다"가 더 중요하다.

## 권장 역할 구조

| 역할 | 주 책임 | 산출물 예시 |
|---|---|---|
| 제품/발표 오너 | 발표 스토리, 데모 시나리오, 우선순위 | 발표 아웃라인, 데모 순서, Q&A 정리 |
| 백엔드 오너 | FastAPI, workflow, 데이터 처리 계약 | route, payload, runtime 흐름 검증 |
| 프론트엔드 오너 | Workbench UI, SSE 표시, approval UX | 화면 동작, 데모 UI 안정성 |
| 데이터/평가 오너 | benchmark, 질문셋, 정량 지표 | confusion matrix, 성공률 표 |
| 문서/검증 오너 | architecture 문서, 운영 문서, 검증 로그 정리 | docs 갱신, 검증 요약, 발표 참고자료 |
| AI 운영 오너 | 프롬프트 관리, AI 사용 정책, 작업 기록 품질 | prompt 템플릿, 사용 가이드, 리스크 목록 |

한 사람이 여러 역할을 겸할 수 있지만, 각 역할의 최종 책임자는 명시해야 한다.

## repository 기준 권장 소유 영역

| 영역 | 1차 오너 | 2차 리뷰어 | 비고 |
|---|---|---|---|
| `backend/app/main.py`, `backend/app/modules/` | 백엔드 오너 | 문서/검증 오너 | public route, status code 영향 큼 |
| `backend/app/orchestration/` | 백엔드 오너 | 제품/발표 오너 | 발표에서 설명 빈도 높음 |
| `frontend/src/app/`, `frontend/src/lib/api.ts` | 프론트엔드 오너 | 백엔드 오너 | SSE/approval 계약 연동 |
| `docs/architecture/` | 문서/검증 오너 | 해당 코드 오너 | 코드 기준 사실 문서 |
| `docs/testing/`, `docs/team/` | 문서/검증 오너 | 제품/발표 오너 | 운영 기준 문서 |
| `docs/reference/` | 제품/발표 오너 | 문서/검증 오너 | 발표 참고자료 |

## 의사결정 규칙

### 코드 변경
- 구현 오너가 초안을 낸다.
- 인접 영역 오너가 계약 파손 여부를 본다.
- 문서/검증 오너가 증거와 기록을 본다.

### 문서 변경
- 문서/검증 오너가 문구 일관성을 본다.
- 코드 기준 문서라면 해당 코드 오너가 사실 여부를 확인한다.
- 발표 영향이 있으면 제품/발표 오너가 최종 표현을 확인한다.

### benchmark/발표 자료 변경
- 데이터/평가 오너가 지표 정의와 산출 근거를 관리한다.
- 제품/발표 오너가 슬라이드 메시지와 연결한다.

## PR 소유권 규칙

PR마다 아래 네 항목을 명시한다.

- 작성자
- 1차 오너
- 필수 리뷰어
- 발표 영향 유무

예시:

```text
작성자: 홍길동
1차 오너: 백엔드 오너
필수 리뷰어: 프론트엔드 오너, 문서/검증 오너
발표 영향: 있음 (SSE approval 설명 변경)
```

## 충돌 방지 규칙

- 같은 파일을 여러 사람이 동시에 크게 수정하지 않는다.
- 공유 파일을 건드리기 전에 누가 최종 통합하는지 정한다.
- 발표 직전에는 신규 구조 변경보다 기존 데모 경로 안정화를 우선한다.

## 발표 주간 운영 권장안

### D-7 ~ D-3
- 기능 보강 가능
- benchmark/문서 정리 병행
- 데모 시나리오 확정 시작

### D-2 ~ D-1
- 큰 구조 변경 금지
- 리허설에서 드러난 버그만 수정
- 발표 문구와 검증 로그 정합성 점검

### 발표 당일
- 담당자별 즉시 대응 범위 명확화
- 데모 실패 시 대체 설명 책임자 지정
- merge보다 발표 안정성을 우선

## 인수인계 최소 템플릿

담당자가 자리를 비우거나 작업을 넘길 때는 아래만 남겨도 충분하다.

```text
현재 상태:
남은 작업:
확인한 명령/로그:
위험 요소:
발표 영향:
```

## 한 줄 결론

역할 분리가 중요한 이유는 일을 나누기 위해서가 아니라, 발표 순간에 "누가 무엇을 책임지고 설명할 수 있는가"를 분명히 하기 위해서다.
