# 역할과 소유권

## 문서 목적

이 문서는 캡스톤 발표 전까지 팀 내 작업 소유권, 리뷰 책임, 발표 반영 책임을 명확히 정리한다.
AI가 초안을 만들 수는 있지만 최종 판단과 외부 설명은 사람 담당자가 맡는다.

## 역할 정의

| 역할 | 책임 |
| --- | --- |
| 구현 담당 | 코드 또는 문서 변경 수행, 1차 검증 |
| 리뷰 담당 | 품질 게이트 확인, 회귀 위험 검토 |
| 발표 담당 | 발표 자료, 데모 흐름, Q&A 대응 메시지 관리 |
| 기록 담당 | benchmark 결과, 검증 로그, 발표 근거 정리 |

## 영역별 기본 소유권

| 영역 | 1차 소유자 역할 | 2차 검토 역할 | 발표 연계 포인트 |
| --- | --- | --- | --- |
| `backend/app/**` | 구현 담당 | 리뷰 담당 | 라우팅, 분석, 오류 처리 설명 |
| `frontend/src/app/**` | 구현 담당 | 리뷰 담당 | Workbench, SSE, approval 데모 |
| `docs/architecture/**` | 구현 담당 또는 문서 담당 | 리뷰 담당 | 시스템 구조 슬라이드 |
| `docs/product/**` | 발표 담당 | 리뷰 담당 | 문제 정의, 범위, 로드맵 |
| `docs/testing/**` | 기록 담당 | 리뷰 담당 | 정량 검증/품질 게이트 설명 |
| `docs/team/**` | 기록 담당 | 발표 담당 | 운영 정책, 협업 방식 설명 |

## 의사결정 규칙

### 혼자 결정 가능한 것

- 오탈자 수정
- 표현 정리
- 근거가 명확한 문서 보강
- 기존 패턴을 따르는 작은 UI/문서 정리

### 리뷰 후 결정할 것

- API 계약 설명 변경
- workflow/status/event 이름 변경
- 발표 메시지에 들어가는 핵심 기술 설명 변경
- benchmark 기준·평가 방식 변경

### 발표 담당 승인까지 필요한 것

- 데모 시나리오 변경
- 발표 슬라이드의 핵심 주장 변경
- 정확도/성능 수치 공개
- 한계점/향후 계획 메시지 수정

## 책임 분리 원칙

1. **작성자와 승인자는 분리한다.**
2. **발표 메시지와 구현 메시지는 같이 맞춘다.**
3. **정량 수치는 기록 담당이 출처를 관리한다.**
4. **문서 전용 변경도 발표 영향이 있으면 발표 담당이 본다.**

## 핸드오프 체크리스트

### 구현 담당 → 리뷰 담당

- [ ] 변경 요약
- [ ] 검증 명령과 결과
- [ ] 영향 범위
- [ ] 남은 우려 사항

### 리뷰 담당 → 발표 담당

- [ ] 발표 메시지 변경 필요 여부
- [ ] 데모 영향 여부
- [ ] 추가 리허설 필요 여부

### 기록 담당 → 팀 전체

- [ ] benchmark 결과 위치
- [ ] 최신 검증 로그 위치
- [ ] 발표용 스크린샷/캡처 위치

## 발표 당일 운영

- 발표 담당: 시나리오 진행과 메시지 책임
- 구현 담당: 데모 장애 대응
- 리뷰 담당: 질의응답 시 근거 파일/명령 지원
- 기록 담당: benchmark/검증 수치 근거 즉시 제시
