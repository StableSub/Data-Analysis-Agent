# 리뷰와 검증 운영 가이드

## 문서 목적

이 문서는 팀이 AI 도움을 받아 만든 변경을 어떻게 리뷰하고, 어떤 증거를 남기고, 언제 발표 자료에 반영할 수 있는지 정리한다.

## 리뷰 목표

- 코드/문서 변경이 실제 저장소 상태와 일치하는지 확인한다.
- 검증 가능한 주장만 merge하고 발표에 사용한다.
- 데모 실패 가능성이 큰 변경을 조기에 걸러낸다.

## 리뷰 역할

| 역할 | 핵심 책임 |
| --- | --- |
| 작성자 | 변경 구현, 자체 검증, 증빙 준비 |
| 리뷰어 | 주장-근거 일치, 회귀 위험, 발표 영향 검토 |
| 발표 담당 | 발표 스크립트·슬라이드·데모 동선 반영 여부 확인 |

자세한 ownership은 `./roles-and-ownership.md`를 따른다.

## 리뷰 절차

### 1. 작성자 제출

작성자는 PR 또는 작업 기록에 아래 내용을 남긴다.

- 변경 배경
- 변경 파일 목록
- 검증 명령과 결과
- 남은 리스크
- 발표 영향도

### 2. 리뷰어 확인

리뷰어는 아래 순서로 본다.

1. 요청 범위와 diff 범위가 맞는지
2. 검증 명령이 실제로 관련성이 있는지
3. 문서 설명이 코드와 맞는지
4. 데모/발표 메시지에 문제를 만들지 않는지

### 3. 발표 반영 결정

발표에 직접 영향을 주는 변경은 아래 중 하나로 분류한다.

- **즉시 반영 가능**: 검증 완료, 데모 영향 낮음
- **리허설 후 반영**: 승인 흐름/SSE/UI 변동 포함
- **발표 후 반영**: 리스크가 크거나 정량 근거가 부족함

## 실전 리뷰 체크리스트

### 공통

- [ ] 제목과 본문이 변경 이유를 설명한다.
- [ ] 검증 로그가 있다.
- [ ] 과장된 표현이 없다.
- [ ] TODO/placeholder가 남아 있지 않다.

### 문서 리뷰

- [ ] 코드 경로가 실제 존재한다.
- [ ] route/event/status 이름이 실제 코드와 일치한다.
- [ ] 발표에서 읽어도 오해 없는 문장이다.

### 코드 리뷰

- [ ] 기존 동작을 깨는 범위가 아닌지 확인했다.
- [ ] 관련 테스트/빌드 결과가 있다.
- [ ] 에러/승인/빈 데이터 경로를 생각했다.

### 발표 영향 리뷰

- [ ] 데모 시나리오 변경 여부를 표시했다.
- [ ] 아키텍처 슬라이드 수정 필요 여부를 표시했다.
- [ ] benchmark/정확도 수치 반영 여부를 표시했다.

## 검증 증빙 묶음

발표 전에는 아래 다섯 가지가 한 곳에서 보여야 한다.

1. 변경 파일 목록
2. 마지막 검증 명령
3. PASS/FAIL 결과
4. 데모 시나리오 영향도
5. 남은 리스크

예시:

```text
Files:
- docs/team/ai-coding-policy.md
- docs/testing/quality-gates.md

Verification:
- PASS | git diff --check
- PASS | PYTHONPATH=. pytest -q backend/tests/test_architecture_docs.py backend/tests/test_docs_harness.py

Demo impact:
- 없음 (문서 전용)

Risk:
- benchmark는 운영 규칙만 정의되었고 자동 runner는 아직 없음
```

## PR 리뷰 게이트

아래 항목이 없으면 승인하지 않는다.

- 검증 명령
- 검증 결과
- 남은 리스크
- 소유자 또는 리뷰 책임자

## 발표 리허설 전 체크

- [ ] 발표자가 변경 내용을 30초 안에 설명할 수 있다.
- [ ] “어떻게 검증했나요?” 질문에 파일 경로와 명령으로 답할 수 있다.
- [ ] 실패/예외 흐름이 발표 메시지와 충돌하지 않는다.
- [ ] 문서와 실제 화면/로그가 서로 모순되지 않는다.
