# 리뷰와 검증 운영 가이드

## 목적

이 문서는 AI 보조 개발 환경에서 사람이 어떤 순서로 리뷰하고 검증해야 하는지 정리한다.
핵심은 "작성자 1회 확인"이 아니라 "다른 사람이 재설명하고 재검증할 수 있는 상태"로 만드는 것이다.

## 기본 흐름

1. 작성자가 변경 목적과 범위를 요약한다.
2. 작성자가 1차 검증을 수행한다.
3. 리뷰어가 diff와 검증 로그를 읽는다.
4. 필요하면 로컬 재검증 또는 데모 흐름 재현을 한다.
5. 발표 영향까지 확인한 뒤 승인/보류를 결정한다.

## 작성자 책임

작성자는 "코드를 만들 사람"이 아니라 "변경을 증명할 사람"이다.

필수 산출물:

- 변경 파일 목록
- 변경 이유
- 검증 명령과 결과
- 미검증 영역
- 발표/데모 영향 여부

작성자 질문 체크리스트:

- [ ] 이 변경을 30초 안에 설명할 수 있는가?
- [ ] 실패했을 때 어디를 되돌리면 되는가?
- [ ] 문서/발표 내용이 함께 바뀌어야 하는가?
- [ ] 내가 실행하지 않은 검증을 암묵적으로 통과한 것처럼 말하고 있지 않은가?

## 리뷰어 책임

리뷰어는 스타일 교정자보다 리스크 탐지자에 가깝다.

우선 확인 순서:

1. 주장과 증거가 맞는가
2. scope가 합리적인가
3. runtime 계약/API/SSE/approval 흐름이 깨지지 않는가
4. 발표용 설명과 충돌하지 않는가
5. 지금 merge해도 복구 가능한가

리뷰어 체크리스트:

- [ ] 커밋/PR 제목이 왜 필요한 변경인지 설명한다.
- [ ] 검증 로그가 실제 명령과 연결된다.
- [ ] 변경 범위가 불필요하게 넓지 않다.
- [ ] 문서 drift 또는 용어 drift가 없다.
- [ ] 발표 당일 질문받을 만한 약점이 드러나면 코멘트로 남겼다.

## PR 승인 게이트

다음 네 조건을 모두 만족해야 승인 가능 상태로 본다.

### 1. 목적이 분명하다
- 문제 또는 필요가 한 문장으로 설명된다.

### 2. 증거가 있다
- 자동 검증 또는 수동 재현 기록이 있다.

### 3. 영향 범위를 안다
- 어떤 파일/기능/문서/데모에 영향 있는지 적혀 있다.

### 4. 남은 리스크가 드러난다
- 미완료, 미측정, 미검증 항목을 숨기지 않았다.

## 문서 변경 리뷰 포인트

문서 PR은 "읽기 좋다"보다 "사실과 맞다"가 먼저다.

- [ ] 코드 기준 문서면 구현 경로가 정확하다.
- [ ] 발표 자료에 재사용해도 오해가 없다.
- [ ] 미래 계획과 현재 구현을 섞어 쓰지 않았다.
- [ ] placeholder, 과장 표현, 근거 없는 정량 표현이 없다.

## 데모/발표 관점 검증

아래 항목 중 하나라도 "아니오"면 발표용 merge를 보류한다.

- [ ] 발표자가 이 변경을 그림 없이 말로 설명할 수 있는가?
- [ ] 데모 중 실패했을 때 대체 설명이 가능한가?
- [ ] 심사위원이 "근거가 뭐냐"고 물었을 때 로그/문서/테스트를 보여줄 수 있는가?
- [ ] 이 변경이 팀이 주장하는 차별점과 일치하는가?

## 코멘트 작성 규칙

리뷰 코멘트는 아래 네 가지 형태 중 하나로 남긴다.

- **Must fix**: merge 전 반드시 해결
- **Should fix**: 가능하면 이번 PR에서 해결
- **Question**: 의도/근거 확인 필요
- **Presentation risk**: 발표에서는 취약할 수 있는 지점

이렇게 구분해야 일정이 촉박할 때도 우선순위가 무너지지 않는다.

## 최종 승인 문구 예시

```text
승인 조건 충족:
- 목적 명확
- 검증 로그 확인
- 발표 영향 확인
- 남은 리스크 문서화 완료
```

## 보류 문구 예시

```text
보류 사유:
- 실행 증거 부족
- 문서/코드 불일치
- 데모 경로 영향 불명확
- 발표 질의응답 리스크 큼
```

## 운영 원칙 한 줄 요약

좋은 리뷰는 "보기 좋게 고치는 것"이 아니라 "발표 자리에서도 방어 가능한 상태로 만드는 것"이다.
