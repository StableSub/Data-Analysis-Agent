# 에이전트 프롬프트 플레이북

## 목적

이 문서는 팀이 AI 코딩 에이전트에게 일을 맡길 때 프롬프트 품질을 일정하게 유지하기 위한 실전 가이드다.
AGENTS.md 같은 런타임 제어 문서를 대체하려는 것이 아니라, 사람이 어떤 식으로 요청해야 더 좋은 결과와 더 적은 재작업을 얻는지 정리한다.

## 기본 원칙

- 좋은 프롬프트는 장문의 추상 설명보다 **목표, 범위, 금지사항, 검증 기준**이 명확하다.
- 이 repo에서는 특히 **존재하지 않는 스크립트/계약을 지어내지 않게 하는 제약**이 중요하다.
- 발표 직전 작업일수록 "새 기능 추가"보다 "문서화, 검증, 데모 안전성"을 분명히 적는다.

## 항상 포함할 5가지

1. **목표** — 무엇을 끝내야 하는가
2. **범위** — 어떤 파일/영역만 다룰 것인가
3. **금지사항** — 무엇을 하면 안 되는가
4. **검증** — 어떤 명령/수동 확인으로 끝낼 것인가
5. **보고 형식** — 결과를 어떻게 요약할 것인가

## 좋은 프롬프트 템플릿

```text
목표:
범위:
금지사항:
완료 조건:
검증 명령:
보고 형식:
```

## 상황별 템플릿

## 1. 문서 작업 요청

```text
목표: 캡스톤 발표용 운영 문서를 한국어로 작성/수정한다.
범위: docs/team/*.md, docs/testing/*.md만 수정한다.
금지사항: 코드 수정 금지, docs/README.md 수정 금지, 존재하지 않는 CI/lint/check-types 언급 금지.
완료 조건: 문서가 현재 repo 사실과 모순되지 않고 체크리스트/발표 관점을 포함한다.
검증 명령: PYTHONPATH=. pytest -q backend/tests/test_architecture_docs.py backend/tests/test_docs_harness.py
보고 형식: 변경 파일, 핵심 추가 내용, 검증 결과, 남은 리스크.
```

## 2. 아키텍처 문서 갱신 요청

```text
목표: runtime 코드와 문서 drift를 맞춘다.
범위: docs/architecture/...와 관련 코드 참조 확인.
금지사항: 미래 계획을 현재 구현처럼 쓰지 말 것.
완료 조건: route, payload, SSE event, approval contract 설명이 코드와 일치한다.
검증 명령: PYTHONPATH=. pytest -q backend/tests/test_architecture_docs.py backend/tests/test_docs_harness.py
```

## 3. 버그 수정 요청

```text
목표: 특정 재현 가능한 버그를 수정한다.
범위: 관련 파일만 최소 수정한다.
금지사항: 무관한 리팩터링, 신규 의존성 추가 금지.
완료 조건: 재현 단계 기준으로 버그가 사라지고 관련 검증이 통과한다.
검증 명령: <실제 존재하는 pytest/build 명령만 기입>
재현 정보: 입력 데이터, 질문, 기대 동작, 실제 동작.
```

## 4. 발표 직전 안정화 요청

```text
목표: 데모 성공률을 높이기 위한 안정화만 수행한다.
범위: 데모 경로와 직접 관련된 파일만 수정한다.
금지사항: 큰 구조 변경, 신규 기능 추가, 설명 불가능한 최적화 금지.
완료 조건: 데모 시나리오가 1회 이상 재현되고 문서/설명이 일치한다.
검증: build, 관련 pytest, 수동 데모 확인.
```

## 피해야 할 나쁜 프롬프트

### 나쁜 예 1
"전체적으로 좀 더 좋게 고쳐줘"
- 문제: 목표와 범위가 없다.

### 나쁜 예 2
"알아서 lint/typecheck까지 다 통과시켜줘"
- 문제: repo에 없는 스크립트를 가정하게 만든다.

### 나쁜 예 3
"발표용으로 멋지게 설명도 추가해줘"
- 문제: 현재 구현과 미래 계획을 섞어 과장할 위험이 크다.

## 추가 제약 문구 예시

필요할 때 아래 문구를 그대로 붙여도 좋다.

- "현재 repo에 없는 명령이나 스크립트를 지어내지 마라."
- "검증 결과는 실제 실행한 명령만 적어라."
- "코드 기준 문서이므로 관련 구현 파일을 먼저 확인하라."
- "발표 슬라이드에서 그대로 읽어도 되는 수준으로 과장 없이 써라."
- "scope 밖 파일은 건드리지 마라."

## 리뷰 친화적 요청 작성법

프롬프트에 아래를 넣으면 결과 품질이 눈에 띄게 좋아진다.

- 변경 파일을 명시한다.
- 제외 파일을 명시한다.
- 성공 기준을 PASS/FAIL로 적는다.
- 보고 형식을 미리 준다.
- 발표 영향 여부를 적는다.

## 팀 내부 권장 운영

- 작업 시작 전, 프롬프트를 슬랙/노션/회의 노트에 한 번 남긴다.
- 반복 작업은 템플릿을 복사해 재사용한다.
- 잘 나온 프롬프트와 실패한 프롬프트를 함께 축적한다.

## 발표 관점 팁

발표 직전 프롬프트에는 아래 중 최소 하나를 꼭 포함한다.

- "심사위원 질문에 답할 수 있도록 근거를 남겨라"
- "데모 실패 시 대체 설명 포인트를 적어라"
- "현재 구현과 향후 계획을 분리해서 써라"

## 한 줄 결론

에이전트 성능 차이보다 더 큰 차이를 만드는 것은, 팀이 얼마나 구체적이고 검증 가능한 요청을 주는가이다.
