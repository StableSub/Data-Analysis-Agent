# 에이전트 프롬프트 플레이북

## 문서 목적

이 문서는 팀원이 AI 도구에 작업을 맡길 때 사용할 수 있는 실전 프롬프트 패턴을 정리한다.
루트 `AGENTS.md`를 대체하지 않으며, 사람 팀원이 더 일관된 입력을 주기 위한 보조 가이드다.

## 기본 원칙

- 요청에는 **목표**, **범위**, **금지 사항**, **검증 기준**을 같이 적는다.
- “좋게 해줘”보다 파일 경로와 기대 결과를 구체적으로 적는다.
- 발표 전에는 근거 없는 추정 대신 “파일 경로/명령 포함”을 명시한다.

## 공통 프롬프트 템플릿

```text
목표: <무엇을 끝내야 하는가>
범위: <수정 가능한 파일/디렉터리>
금지: <건드리면 안 되는 파일/주장/행동>
검증: <반드시 실행할 명령 또는 확인 항목>
결과 형식: <원하는 보고 형식>
```

## 상황별 템플릿

### 1. 문서 정리 요청

```text
목표: docs/team 아래 운영 문서를 한국어로 작성
범위: docs/team/*.md, docs/testing/*.md
금지: docs/README.md 수정 금지, 존재하지 않는 CI 언급 금지
검증: git diff --check, docs harness 실행
결과 형식: 변경 파일 + 검증 결과 + 남은 리스크
```

### 2. 아키텍처 설명 보강 요청

```text
목표: 현재 코드 기준으로 SSE 이벤트 계약 문서를 작성
범위: docs/architecture/system/api-sse-error-contract.md
금지: 계획 중 기능을 현재 동작처럼 설명 금지
검증: 코드 경로 존재 확인, docs harness 실행
결과 형식: 기준 코드 경로, 이벤트 표, 프론트엔드 소비 방식 포함
```

### 3. 코드 수정 요청

```text
목표: 특정 버그 수정
범위: 관련 모듈과 테스트 파일만
금지: 무관한 리팩터링 금지, 새 dependency 추가 금지
검증: 관련 pytest, 필요 시 npm build
결과 형식: 원인, 수정 내용, 검증 로그
```

### 4. 발표 근거 정리 요청

```text
목표: 발표 슬라이드에 넣을 근거 문장을 저장소 기준으로 정리
범위: docs/product, docs/architecture, docs/testing
금지: benchmark 수치 추정 금지
검증: 각 주장 옆에 파일 경로 포함
결과 형식: 주장 / 근거 파일 / 발표용 한 줄 요약
```

## 좋은 프롬프트 예시

- “`backend/app/modules/chat/router.py`와 `frontend/src/app/hooks/useAnalysisPipeline.ts` 기준으로 SSE 이벤트 문서를 작성하고, docs harness까지 확인해줘.”
- “`docs/testing/benchmark-spec.md`를 만들되 자동 benchmark가 없다는 점을 명시하고 발표용 수동 측정 규칙을 정리해줘.”

## 나쁜 프롬프트 예시

- “대충 문서 좀 정리해줘.”
- “없는 테스트도 다 통과했다고 맞춰줘.”
- “발표에 좋아 보이게 성능 수치 만들어줘.”

## 프롬프트 작성 체크리스트

- [ ] 파일 경로를 넣었는가?
- [ ] 하지 말아야 할 일을 적었는가?
- [ ] 검증 기준을 적었는가?
- [ ] 결과 보고 형식을 적었는가?

## 발표 직전 전용 문구

발표 전에는 아래 문구를 붙이는 것이 안전하다.

```text
과장 없이 현재 저장소 기준으로만 작성하고,
검증하지 않은 성능/정확도 주장은 넣지 말고,
실행한 명령과 파일 경로를 함께 보고해줘.
```

## 후속 운영 팁

- 반복해서 잘 쓰는 프롬프트는 팀 노션이나 발표 준비 문서로 재사용한다.
- 결과가 좋았던 프롬프트도 범위와 금지 조건이 빠져 있으면 다음 작업에서 다시 다듬는다.
- AGENTS.md, product 문서, architecture 문서를 먼저 읽히는 요청이 가장 안정적이다.
