# Docs

프로젝트 문서는 역할에 따라 세 계층으로 분리한다.

## 계층 구조

| 폴더 | 대상 | 내용 |
|------|------|------|
| `architecture/` | 개발자 | 코드를 이해하기 위한 구현 참조 문서 |
| `planning/` | 팀 전체 | 무엇을 왜 만드는지 기획·결정 문서 |
| `reference/` | 발표·외부 | 제안서, 분석 결과 등 외부 입력 문서 |

## 디렉터리 구조

```text
docs/
├── architecture/          # 구현 참조 문서
│   ├── ai-agent/          # AI Agent 역할, 실행 흐름, 프롬프트, 로깅
│   ├── system/            # 시스템 흐름, 구조, API, 용어
│   ├── components/        # 워크플로우 컴포넌트별 상세
│   ├── request-lifecycle.md
│   └── shared-state.md
│
├── planning/              # 기획 문서 (노션 마이그레이션)
│   ├── Product/           # 문제 정의, UX, PRD, 시나리오
│   ├── Data/              # 데이터 구조와 데이터베이스
│   ├── Infra-Ops/         # 실행 환경, 배포, 운영, 장애 대응
│   └── Decision-Log/      # 구조 결정 기록과 변경 이력
│
└── reference/             # 외부 입력 문서
    ├── 산학캡스톤_프로젝트_제안서.pdf
    └── capstone-presentation-analysis.md
```

## 문서 제목 변경 기준

- 너무 짧거나 약어만 있는 제목은 의미가 바로 드러나게 확장한다.
- 영어/한글이 섞인 제목은 한글 중심으로 통일한다.
- `개요`, `구조`, `설계`, `전략`, `기록`처럼 문서 성격이 드러나는 표현을 우선 사용한다.
