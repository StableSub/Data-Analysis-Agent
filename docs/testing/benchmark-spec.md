# 벤치마크 스펙

## 목적

이 문서는 캡스톤 발표 전에 확보해야 할 최소 정량 평가 항목을 정의한다.
현재 repository에는 완성된 벤치마크 파이프라인이 없으므로, 이 문서는 "이미 있는 결과 보고서"가 아니라 "팀이 동일한 방식으로 측정하기 위한 운영 기준"이다.

## 왜 필요한가

현재 프로젝트의 강점은 역할 분리된 AI workflow와 approval 기반 상호작용이다.
하지만 발표에서는 구조 설명만으로 충분하지 않다. 최소한 아래 질문에는 숫자로 답할 수 있어야 한다.

- 라우팅이 얼마나 자주 맞는가?
- 분석 실행은 얼마나 자주 끝까지 성공하는가?
- RAG는 근거를 얼마나 잘 찾는가?
- approval이 필요한 흐름은 데모에서 얼마나 안정적으로 재현되는가?

## 벤치마크 원칙

- 발표 전에 재현 가능한 소규모 질문셋이라도 먼저 고정한다.
- 숫자는 적어도 근거가 있어야 한다.
- 측정 불가능한 항목은 "정성 평가"로 분리한다.
- 한 번 좋은 결과가 아니라 반복 가능한 결과를 남긴다.

## 최소 측정 항목

## 1. 라우팅 정확도

대상:
- 일반 질문
- 데이터셋 기반 분석 질문
- RAG 질문
- approval이 개입되는 질문

기록 항목:
- 질문 ID
- 기대 route
- 실제 route
- 일치 여부
- 오분류 이유

최소 목표:
- 20~30개 질문
- confusion matrix 작성 가능 수준

## 2. 분석 실행 성공률

대상:
- 선택된 데이터셋 기준 대표 분석 질문 10~15개

기록 항목:
- 질문 ID
- source_id 또는 데이터셋 이름
- 첫 시도 성공 여부
- 재시도 후 성공 여부
- 실패 stage
- 결과 요약 품질 메모

권장 지표:
- first-pass success rate
- final success rate
- repair success rate

## 3. 답변 정확도/유용성

완전한 자동 채점이 어려우므로, 발표 전에는 소규모 human review로 운영한다.

평가 축:
- 질문 의도 충족 여부
- 핵심 수치/결론 일치 여부
- 근거 설명 가능성
- 과장/환각 여부

평가 등급 예시:
- 2점: 정확하고 바로 사용 가능
- 1점: 부분적으로 유용하지만 보완 필요
- 0점: 부정확하거나 사용 곤란

## 4. RAG 근거 검색 품질

대상:
- guideline 또는 문서 기반 질문 10개 내외

기록 항목:
- 질문 ID
- 기대 근거 문서
- 검색된 chunk 수
- 상위 chunk 적합 여부
- 최종 답변에 근거 반영 여부

권장 지표:
- hit@k
- 정답 근거 포함률
- no-evidence 처리 비율

## 5. approval 흐름 재현성

대상:
- preprocess approval
- visualization approval
- report draft review

기록 항목:
- 시나리오 ID
- 어떤 approval 단계인지
- `approval_required` 이벤트 확인 여부
- resume 후 완료 여부
- 실패 시 증상

이 항목은 UX 안정성과 발표 데모 안전성에 직접 연결된다.

## 데이터셋 구성 권장안

최소 두 그룹으로 나눈다.

### A. 데모용 고정 데이터셋
- 발표에서 실제 보여줄 데이터
- 질문셋을 사전에 고정
- 성공 경로 재현성 우선

### B. 검증용 보조 데이터셋
- edge case 확인용
- 결측치, 이상치, 시계열, 범주형 혼합 등 포함

## 기록 포맷

CSV 또는 스프레드시트에 아래 컬럼을 통일해서 남긴다.

```text
benchmark_id, category, dataset, question, expected_behavior, actual_behavior, pass_fail, notes
```

세부 측정이 필요하면 추가 컬럼을 붙인다.

예시:

```text
R-01,routing,titanic,"생존률을 보여줘",analysis,analysis,PASS,
A-03,analysis,sales_q1,"월별 매출 추세를 비교해줘",successful_run,failed_after_retry,FAIL,column mismatch
P-02,approval,customer_churn,"결측치 전처리 후 분석해줘",approval_resume_success,approval_resume_success,PASS,
```

## 운영 절차

1. 질문셋 고정
2. 데이터셋 버전 고정
3. 실행 담당자와 채점 담당자 분리
4. 결과 수집
5. 실패 유형 묶기
6. 발표용 핵심 표/그래프 2~3개만 추리기

## 발표 자료로 연결할 최소 산출물

- 라우팅 confusion matrix 1장
- 분석 성공률 표 1장
- approval/resume 재현 흐름 1장
- 대표 실패와 개선 계획 1장

## 주의사항

- 아직 없는 자동 하네스를 이미 갖춘 것처럼 설명하지 않는다.
- 샘플 수가 적으면 적다고 밝히고, 대신 측정 기준이 일관되다고 설명한다.
- 숫자가 좋지 않아도 숨기지 말고 개선 우선순위와 함께 제시한다.

## 완료 체크리스트

- [ ] 질문셋이 고정되었다.
- [ ] 데이터셋 버전이 고정되었다.
- [ ] 라우팅/분석/RAG/approval 항목이 모두 최소 1회 이상 측정되었다.
- [ ] PASS/FAIL 기준이 문서로 남아 있다.
- [ ] 발표 슬라이드에 들어갈 표/그래프 후보를 골랐다.
